# Milestone 1 — Authentication
### FanoutFeed · `milestone-1-auth`

---

## Goal

Replace the `?author_id=` query parameter pattern with verified JWT-based identity.
Every user action — posting, following, connecting via WebSocket — is now tied to a
real authenticated account that cannot be forged by another client.

---

## Previous architecture and its limitation

At the end of Milestone 0.5, the system had persistent storage (PostgreSQL + Redis)
and a working event-driven fanout pipeline, but no identity layer:

```
POST /posts?author_id=alice
```

Any client could set `author_id` to any value. There was no way to know if the
caller was actually Alice. This made every user-specific feature impossible to build
correctly: rate limiting, moderation, bookmarks, notifications, and DMs all require
a verified identity. There was also no concept of a session, so there was no way to
log someone out or revoke access.

---

## Why this milestone comes before more features

Auth cannot be retrofitted. Every route that currently accepts `?author_id=` as a
parameter would need to be rewritten. Every consumer that trusts the event payload's
`author_id` would need to be re-evaluated. Every test would break. The longer auth
is deferred, the more expensive it becomes. Adding it now — before rate limiting,
DMs, notifications, or any other user-specific feature — is the correct sequencing.

---

## Design decisions

### Two tokens, not one

A single long-lived token is a common shortcut that creates a serious problem: there
is nothing to revoke. If a token is stolen or a user wants to log out of a device,
the token remains valid until it expires.

The two-token model solves this:

```
Access token  (15 min, stateless JWT)
  → verified by signature alone, no DB lookup needed
  → cannot be revoked mid-life; expires naturally
  → safe to attach to every request because it expires quickly

Refresh token (30 days, opaque random string)
  → stored in the database as a SHA-256 hash
  → used only to obtain a new access token
  → can be revoked at any time (logout, suspicious activity)
```

Think of it like a hotel key card (access token) and the front desk record
(refresh token). The card gets you through doors without bothering the desk on
every swipe. But if you lose it, the desk can deactivate the record and issue
a new card.

### Refresh token rotation

Every time a refresh token is used to get a new access token, the old refresh token
is invalidated and a new one is issued. The client must store the new token for the
next refresh cycle.

This creates a theft-detection mechanism: if a refresh token is stolen and used by
an attacker before the legitimate user makes their next request, the legitimate
user's next refresh attempt will find the token already consumed. The server detects
this as a potential theft event and revokes **all** sessions for that user — forcing
a full re-login on all devices.

### Why SHA-256 for refresh tokens, not bcrypt

Passwords are hashed with bcrypt because they are user-chosen and potentially
guessable. Bcrypt's deliberate slowness makes brute-force attacks impractical.

Refresh tokens are 64 bytes of cryptographically random data generated by
`secrets.token_urlsafe()`. There is nothing to brute-force — the keyspace is
2^512. SHA-256 is fast and sufficient to protect them at rest. Using bcrypt here
would add ~100ms of hashing time on every `/auth/refresh` call for no security
benefit.

### Why the WebSocket token travels as a query parameter

Browser WebSocket API does not support custom headers. The `Authorization: Bearer`
header pattern used by HTTP endpoints is unavailable when opening a WebSocket
connection from a browser. The standard solution is a short-lived token in the
query string:

```
ws://host/ws/feed?token=<access_token>
```

The access token's 15-minute expiry limits the window of exposure if the URL is
logged or leaked. The WebSocket endpoint validates the token on connection
establishment only — mid-session re-validation would require closing and reopening
the connection, which is deferred to a future milestone.

### Logout semantics

`POST /auth/logout` revokes the refresh token. The access token is NOT blacklisted
— it remains valid until its 15-minute natural expiry.

Blacklisting access tokens requires a DB lookup (or Redis check) on every single
request, which eliminates the primary performance benefit of stateless JWTs. The
15-minute window is an accepted trade-off. In practice, logout on a compromised
account would be followed by a password change, which the attacker's outstanding
access token cannot reflect.

## What was built

### New endpoints

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/auth/register` | None | Create account, return token pair |
| `POST` | `/auth/login` | None | Verify credentials, return token pair |
| `POST` | `/auth/refresh` | Refresh token in body | Rotate refresh token, return new pair |
| `POST` | `/auth/logout` | Refresh token in body | Revoke refresh token |

### Changed endpoints

| Before | After | Change |
|---|---|---|
| `POST /posts?author_id=alice` | `POST /posts` + Bearer token | Author from JWT |
| `POST /users/:id/follow/:tid` | `POST /me/follow/:tid` | User from JWT |
| `DELETE /users/:id/follow/:tid` | `DELETE /me/follow/:tid` | User from JWT |
| `GET /users/:id/following` | `GET /me/following` | User from JWT |
| `WS /ws/:user_id` | `WS /ws/feed?token=` | Token in query param |

### Unchanged (still public)

- `GET /users` — user list
- `GET /users/:id/following` — view any user's following list
- `GET /timeline/:id` — view any user's timeline
- `WS /ws/events` — system event broadcast (debug panel)

## Database schema changes

```sql
-- Added to users table
ALTER TABLE users ADD COLUMN IF NOT EXISTS password_hash TEXT;
-- NULL for seed users until first startup assigns a default password

-- New table
CREATE TABLE IF NOT EXISTS refresh_tokens (
    id          TEXT             PRIMARY KEY,
    user_id     TEXT             NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash  TEXT             NOT NULL UNIQUE,   -- SHA-256 of the raw token
    expires_at  DOUBLE PRECISION NOT NULL,
    created_at  DOUBLE PRECISION NOT NULL,
    revoked     BOOLEAN          NOT NULL DEFAULT FALSE
);

CREATE INDEX IF NOT EXISTS idx_refresh_tokens_user ON refresh_tokens(user_id);
```

## Request flow comparison

### Before (Milestone 0.5)

```
Client                                                       FastAPI
  │                                                             │
  ├─ POST /posts?author_id=alice ──────────────────────────────►│
  │                          │  author_id taken at face value   │
  │                          │  (anyone could send any user_id) │
  │◄─────────────────── { post_id } ────────────────────────────│
```

### After (Milestone 1)

```
Client                    FastAPI                         PostgreSQL
  │                          │                                   │
  ├─ POST /auth/login ──────►│                                   │
  │  { username, password }  ├─ get_user_with_password() ──────► │
  │                          │◄─────────── { id, name, hash } ───│
  │                          │  bcrypt.verify(password, hash)    │
  │◄── { access_token,       │                                   │
  │      refresh_token } ────│  store_refresh_token() ──────────►│
  │                          │                                   │
  │  (later)                 │                                   │
  │                          │                                   │
  ├─ POST /posts ───────────►│                                   │
  │  Authorization: Bearer   │  decode_access_token()            │
  │  <access_token>          │  (no DB hit — signature only)     │
  │                          │  author_id = payload["sub"]       │
  │◄── { post_id } ──────────│                                   │
```

### Refresh flow

```
Client                    FastAPI              PostgreSQL
  │                          │                     │
  ├─ POST /auth/refresh ─────►│                     │
  │  { refresh_token: "..." } ├─ get_refresh_token(sha256(raw)) ►│
  │                          │◄── { user_id, revoked, expires } ─│
  │                          │  revoked? → 401 + revoke all      │
  │                          │  expired? → 401                   │
  │                          │  ok → revoke old token ──────────►│
  │                          │       store new token ───────────►│
  │◄── { access_token,       │                     │
  │      refresh_token } ────│                     │
```

## Seed user defaults

The four seed users (alice, bob, carol, dave) are assigned `password123` on first
server startup via `db.set_password_hash()`. This function only updates rows where
`password_hash IS NULL`, so existing passwords are never overwritten on subsequent
restarts.

To log in immediately after pulling this milestone:

```
username: alice   password: password123
username: bob     password: password123
username: carol   password: password123
username: dave    password: password123
```

Add `JWT_SECRET` to your `.env` before starting (generate one):

```bash
python -c "import secrets; print(secrets.token_hex(32))"
```

## Remaining limitations

**Access tokens are not revocable before expiry.**
Logout revokes the refresh token, but the 15-minute access token window remains
open. For most use cases this is acceptable. If immediate revocation is required
(e.g., account suspension), a Redis-based token blocklist would be needed — at the
cost of one cache lookup per request.

**No email or multi-factor auth.**
Username and password only. Email verification and MFA are product concerns that
sit above the current architectural scope.

**Single device per account (last login wins).**
The `ConnectionManager` stores one WebSocket per `user_id`. If the same user logs
in from two browsers simultaneously, the second connection overwrites the first.
Multi-device support requires storing a list of connections per user, which is a
natural extension but deferred.

**Password reset not implemented.**
Without email delivery, there is no self-service password recovery. A future
milestone that adds email (for notification digests or similar) would be the natural
point to add this.

---

## Scaling ceiling unchanged

Authentication adds constant-time overhead per request (one JWT decode, no DB hit).
The scaling limits identified in Milestone 0.5 are unchanged. This milestone does
not affect throughput, concurrency, or fanout performance.

## Next milestone — Milestone 2: Redis Streams

**Problem:** Redis Pub/Sub is fire-and-forget. If the `listen()` background task
is not running when `bus.publish("PostCreated", ...)` is called — during a server
restart, a `uvicorn --reload` reload, or any task crash — the event is silently
dropped. The post exists in PostgreSQL, but fanout never runs. With authenticated
users who expect their followers' timelines to be updated, this is no longer an
acceptable gap.

**What Redis Streams adds:**
- Message persistence: events survive a consumer restart
- Consumer group acknowledgement: a message is not removed until explicitly ACKed
- `XAUTOCLAIM`: failed (unACKed) messages are retried automatically
- Replay: new consumers can read from any point in the stream history

The migration is contained to `event_bus.py` — `PUBLISH/SUBSCRIBE` replaced by
`XADD/XREADGROUP`. Consumer logic in `consumers.py` does not change.
